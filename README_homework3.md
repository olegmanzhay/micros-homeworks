# Задание 1

## Архитектура решения

Предлагаемое решение объединяет следующие компоненты:

* **Bitbucket** — система контроля версий (Git‑репозитории);
* **Jenkins** — оркестратор CI/CD‑процессов;
* **Yandex Cloud** — облачная инфраструктура;
* **Nexus Registry** — хранилище Docker‑образов;
* **Собственные серверы** — для развёртывания агентов сборки.

## Взаимодействие компонентов

1. **Хранение кода**: каждый сервис имеет собственный репозиторий в Bitbucket.
2. **Триггеры сборок**:
    * автоматическая сборка запускается через Webhook из Bitbucket при push/merge в репозиторий;
    * ручная сборка с параметрами запускается через интерфейс Jenkins.
3. **Оркестрация в Jenkins**:
    * использование Jenkinsfile для описания пайплайна;
    * параметризованные джобы для разных конфигураций сборки;
    * шаблоны пайплайнов через Shared Libraries.
4. **Сборка и упаковка**:
    * агенты Jenkins выполняют сборку в Docker‑контейнерах;
    * используются кастомные Docker‑образы из Nexus Registry;
    * параллельное выполнение шагов сборки и тестов.
5. **Безопасность**:
    * секреты хранятся в Credentials Store Jenkins;
    * доступ контролируется через RBAC Jenkins и права Bitbucket.
6. **Инфраструктура**:
    * Jenkins и агенты развёрнуты в Yandex Cloud;
    * дополнительные агенты могут быть развёрнуты на собственных серверах.

## Реализация требований

| Требование | Реализация                                                                   |
|----------|------------------------------------------------------------------------------|
| Облачная система (Yandex Cloud) | Jenkins и агенты развёрнуты в Yandex Cloud (Compute Instance, Kubernetes)    |
| Система контроля версий Git (Bitbucket) | Каждый сервис — отдельный репозиторий в Bitbucket                            |
| Репозиторий на каждый сервис | Отдельный репозиторий Bitbucket для каждого сервиса                          |
| Запуск сборки по событию из системы контроля версий | Webhook в Bitbucket → триггер джоба в Jenkins                                |
| Запуск сборки по кнопке с параметрами | Параметризованные джобы в Jenkins                                            |
| Привязка настроек к каждой сборке | Конфигурация в Jenkinsfile + параметры джоба                                 |
| Шаблоны для конфигураций сборок | Shared Libraries в Jenkins для переиспользуемых пайплайнов                   |
| Безопасное хранение секретных данных | Credentials Store Jenkins + шифрование секретов                              |
| Несколько конфигураций для сборки из одного репозитория | Параметризованный Jenkinsfile с ветвлением логики                            |
| Кастомные шаги при сборке | Groovy‑скрипты в Jenkinsfile, shell/batch‑команды                            |
| Собственные Docker‑образы для сборки | Хранение образов в Nexus Registry, использование в агентах Jenkins           |
| Развёртывание агентов на собственных серверах | Настройка Jenkins agents на локальных серверах                               |
| Параллельный запуск сборок | Конфигурация Jenkins с несколькими агентами, параллельные stages в пайплайне |
| Параллельный запуск тестов | Параллельное выполнение тестов в разных stages пайплайна                     |

## Настройка интеграции Bitbucket → Jenkins

**Шаг 1.** В Bitbucket:
* создать Webhook на событие `Push` или `Pull Request Merged`;
* указать URL Jenkins Webhook (например, `https://your-jenkins/bitbucket-hook`).

**Шаг 2.** В Jenkins:
* установить плагин **Bitbucket Plugin**;
* настроить проект как **Multibranch Pipeline** или **Pipeline**;
* указать репозиторий Bitbucket и учётные данные.

**Шаг 3.** В Jenkinsfile описать пайплайн:

```groovy
pipeline {
    agent {
        docker {
            image 'your-custom-build-image:latest'
            registryUrl 'https://nexus-registry.example.com'
            registryCredentialsId 'nexus-creds'
        }
    }
    parameters {
        string(name: 'DEPLOY_ENV', defaultValue: 'staging', description: 'Среда развёртывания')
        booleanParam(name: 'RUN_TESTS', defaultValue: true, description: 'Запускать тесты')
    }
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package'
            }
        }
        stage('Test') {
            when { expression { return params.RUN_TESTS } }
            parallel {
                stage('Unit Tests') {
                    steps { sh 'mvn test' }
                }
                stage('Integration Tests') {
                    steps { sh 'mvn verify' }
                }
            }
        }
        stage('Publish') {
            steps {
                script {
                    docker.withRegistry('https://nexus-registry.example.com', 'docker-creds') {
                        def app = docker.build("myapp:${env.BUILD_ID}")
                        app.push()
                    }
                }
            }
        }
    }
}
```
# Задание 2
## Решение для сбора и анализа логов в микросервисной архитектуре (на базе ELK‑стека)


**ELK‑стек** (Elastic Stack) состоит из следующих компонентов:


1. **Filebeat** — легковесный агент сбора логов, устанавливается на каждом хосте.
2. **Logstash** — процессор и трансформер логов (фильтрация, парсинг, обогащение).
3. **Elasticsearch** — распределённое хранилище и поисковый движок.
4. **Kibana** — веб‑интерфейс для поиска, визуализации и анализа логов.

## Архитектура и взаимодействие компонентов

1. **Сбор логов**:
    * Filebeat запускается на всех хостах (виртуальных машинах, контейнерах, Kubernetes‑подах).
    * Собирает логи из `stdout`/`stderr` контейнеров или файлов логов.
    * Гарантированно доставляет логи в Logstash/Elasticsearch (буферизация при потере связи).

2. **Обработка логов**:
    * Logstash принимает логи от Filebeat (через Beats‑протокол).
    * Парсит и структурирует логи (JSON, регулярные выражения).
    * Обогащает данные (добавляет метаданные: хост, сервис, окружение).
    * Отправляет обработанные логи в Elasticsearch.

3. **Хранение и поиск**:
    * Elasticsearch индексирует логи, обеспечивает быстрый полнотекстовый поиск.
    * Реализует ротацию индексов (например, по дням: `logs-2024.05.15`).
    * Обеспечивает отказоустойчивость за счёт репликации данных.

4. **Визуализация и анализ**:
    * Kibana предоставляет веб‑интерфейс для:
        * поиска и фильтрации логов (KQL — Kibana Query Language);
        * создания дашбордов и графиков;
        * сохранения поисковых запросов и предоставления ссылок на них.

## Реализация требований

| Требование | Реализация в ELK‑стеке |
|----------|----------------------|
| Сбор логов в центральное хранилище со всех хостов | Filebeat на каждом хосте → Logstash → Elasticsearch |
| Минимальные требования к приложениям, сбор из stdout | Filebeat читает stdout контейнеров (Docker, Kubernetes) или файлы логов |
| Гарантированная доставка логов | Filebeat буферизует логи локально, повторяет отправку при ошибках |
| Поиск и фильтрация по записям логов | Elasticsearch + Kibana (KQL, фильтры по полям, временным интервалам) |
| Пользовательский интерфейс с доступом для разработчиков | Kibana: веб‑интерфейс с ролевой моделью доступа (RBAC) |
| Возможность дать ссылку на сохранённый поиск | В Kibana можно сохранить поисковый запрос как «Saved Search» и поделиться ссылкой |

## Настройка компонентов


### 1. Filebeat (на каждом хосте)

Конфигурация (`filebeat.yml`):
```yaml
filebeat.inputs:
- type: container
  paths:
    - '/var/lib/docker/containers/*/*.log'
  processors:
    - add_docker_metadata: ~

output.logstash:
  hosts: ['logstash-server:5044']
```


# Задание 3
# Решение для мониторинга микросервисной архитектуры (на базе Prometheus + Grafana)

## Архитектура решения

Решение объединяет следующие компоненты:

* **Kubernetes** — оркестратор контейнеров, источник метрик сервисов и узлов.
* **Prometheus** — система сбора, хранения и обработки метрик.
* **Node Exporter** — агент для сбора метрик с Linux‑серверов.
* **Grafana** — веб‑интерфейс для визуализации и анализа данных.
* **kube‑state‑metrics** — компонент для сбора метрик состояния объектов Kubernetes (Pods, Deployments и т. д.).
* **cAdvisor** — компонент Kubernetes, собирающий метрики контейнеров (CPU, RAM, сеть, диск).

## Взаимодействие компонентов

1. **Сбор метрик**:
    * **Node Exporter** запускается на каждом Linux‑сервере, собирает системные метрики (CPU, RAM, HDD, Network) и отдаёт их по HTTP.
    * В Kubernetes **cAdvisor** собирает метрики контейнеров, а **kube‑state‑metrics** — метрики объектов кластера.
2. **Сбор и хранение**:
    * **Prometheus** периодически (по расписанию) забирает метрики у всех экспортеров (Node Exporter, cAdvisor, kube‑state‑metrics).
    * Хранит данные в виде временных рядов (time‑series) с метками (labels).
3. **Визуализация**:
    * **Grafana** подключается к Prometheus как источнику данных.
    * Отображает метрики в виде графиков, таблиц, дашбордов.
    * Позволяет создавать запросы на языке PromQL, агрегировать данные, настраивать оповещения.

## Реализация требований

| Требование | Реализация |
|----------|-----------|
| Сбор метрик со всех хостов | Node Exporter на Linux‑серверах; cAdvisor и kube‑state‑metrics в Kubernetes |
| Сбор метрик состояния ресурсов хостов (CPU, RAM, HDD, Network) | Node Exporter собирает системные метрики с Linux‑хостов |
| Сбор метрик потребляемых ресурсов для каждого сервиса (CPU, RAM, HDD, Network) | cAdvisor в Kubernetes собирает метрики по каждому Pod/контейнеру |
| Сбор метрик, специфичных для каждого сервиса | Сервисы могут выставлять кастомные метрики через HTTP‑эндпоинт `/metrics`, который забирает Prometheus |
| Пользовательский интерфейс с запросами и агрегацией | Grafana с PromQL: запросы, агрегация (sum, rate, avg), группировка по меткам |
| Пользовательский интерфейс с настраиваемыми панелями | Grafana: создание и настройка дашбордов, сохранение шаблонов, совместное использование |
