# Задание 2
### Сравнение брокеров сообщений: Kafka, RabbitMQ, NATS

| Критерий | Apache Kafka | RabbitMQ | NATS |
|--------|------------|----------|------|
| **Поддержка кластеризации** | Да, горизонтальная масштабируемость, репликация партиций, кворумная модель | Да, кластеризация и репликация очередей, но сложнее масштабировать горизонтально | Да, простая кластеризация, поддержка суперкластеров с гео‑маршрутизацией |
| **Хранение на диске** | Да, долговременное хранение в виде журнала (log), сообщения не удаляются после доставки | Да, при настройке долговечных (durable) очередей | Да, через JetStream (persistent storage), по умолчанию — in‑memory |
| **Скорость работы** | Очень высокая пропускная способность (миллионы сообщений/сек), но выше задержка из‑за журналирования | Высокая, но ограничена при большом масштабе; оптимален для средних нагрузок | Очень низкая задержка (sub‑millisecond), высокая пропускная способность, оптимизирован под скорость |
| **Форматы сообщений** | Любые (бинарные, JSON, Avro, Protobuf и т. д.), нет встроенной сериализации | Любые, поддержка AMQP, MQTT, STOMP и др.; гибкая маршрутизация | Любые, но без встроенной сериализации; поддерживает иерархию тем (subject hierarchy) |
| **Разделение прав доступа** | Да, ACL, SASL/SSL аутентификация, интеграция с Kerberos, LDAP | Да, гибкая система прав (vhosts, users, permissions), поддержка SSL | Да, аутентификация по токенам, пользователям, JWT; ACL на уровне subjects |
| **Простота эксплуатации** | Средняя/низкая: требует настройки ZooKeeper (до Kafka 3.3), мониторинга, тонкой настройки параметров | Средняя: проще Kafka, но требует внимания к настройке кластеров и очередей | Высокая: запускается одной командой, минимум настроек, легковесный (написан на Go) |
| **Модель обмена** | Publish‑subscribe (топики и партиции), потоковая обработка | Традиционная очередь (AMQP), гибкая маршрутизация через exchanges | Publish‑subscribe, request‑reply, JetStream для потоковой обработки |
| **Подтверждение доставки** | Коммит оффсетов (смещение), атомарные коммиты для групп | Acknowledgements (подтверждения от потребителей), надёжная доставка | Acknowledgements через JetStream, гарантия доставки |
| **Горизонтальное масштабирование** | Отличное, линейное масштабирование за счёт партиций | Ограниченное, сложнее масштабировать без потери производительности | Хорошее, поддержка глобальных суперкластеров |
| **Экосистема и инструменты** | Богатая: Kafka Connect, Kafka Streams, KSQL, мониторинг (Prometheus, Grafana) | Хорошая: плагины, веб‑интерфейс, интеграции | Развивающаяся: JetStream, NATS CLI, но меньше готовых решений |
| **Типичные сценарии** | Большие данные, аналитика в реальном времени, логирование, ETL‑потоки | Микросервисы, задачи с сложной маршрутизацией, RPC, бизнес‑процессы | Микросервисы, IoT, low‑latency приложения, сервис‑сервис взаимодействие |

---

### Обоснованный выбор решения

**Оптимальный выбор: Apache Kafka**

**Обоснование:**

* **Полное соответствие требованиям:**
    * кластеризация с репликацией и высокой доступностью;
    * долговременное хранение сообщений на диске (журнал);
    * очень высокая пропускная способность и масштабируемость;
    * поддержка любых форматов данных;
    * развитая система контроля доступа (ACL, Kerberos, LDAP);
    * зрелая экосистема и обширная документация.
* **Ключевые преимущества:**
    * подходит для больших объёмов данных и потоковой обработки;
    * гарантирует сохранность данных даже при сбоях;
    * позволяет перечитывать сообщения (идемпотентность);
    * богатый набор инструментов для интеграции и мониторинга.

**Альтернативы:**

1. **NATS** — если критична **низкая задержка** и **простота развёртывания**. Подходит для микросервисов и IoT, где не требуется долговременное хранение всех сообщений (или используется JetStream).
2. **RabbitMQ** — если нужна **гибкая маршрутизация** и поддержка множества протоколов (AMQP, MQTT и т. д.). Оптимален для бизнес‑задач с умеренной нагрузкой и сложной логикой маршрутизации.

---

### Краткий вывод

* Выбирайте **Kafka**, если:
    * обрабатываете большие потоки данных;
    * нужна гарантия сохранности и возможность перечитывания;
    * важна интеграция с экосистемой Big Data.
* Выбирайте **NATS**, если:
    * приоритет — минимальная задержка и простота;
    * архитектура микросервисная, нагрузка умеренная.
* Выбирайте **RabbitMQ**, если:
    * требуется сложная маршрутизация сообщений;
    * используете AMQP или другие протоколы;
    * нагрузка не экстремально высокая.
